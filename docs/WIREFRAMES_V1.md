### C4. Readiness 算法设计（`/api/readiness`）

```
输入:
  - 今日 wellness (HRV, RHR, sleep)
  - 今日 TSB
  - 7日 wellness 均值 + 标准差
  - 今日是否有计划训练
  - 计划训练类型（关键课 vs 耐力 vs 恢复）

判断逻辑:
  1. HRV 在 7日均值 ± 1SD 内 → 正常
     HRV < 7日均值 - 1.5SD → 异常低
  2. RHR 在 7日均值 ± 3bpm 内 → 正常
     RHR > 7日均值 + 5bpm → 异常高
  3. 睡眠 >= 6h → 正常, < 5h → 差
  4. TSB > 0 → 恢复充分, -5~0 → 可训练, -15~-5 → 疲劳, < -15 → 高风险

输出矩阵:
  - 全部正常 + 计划关键课 → "可执行关键课" (高置信度)
  - 全部正常 + 无计划     → "可正常训练" (高置信度)
  - 1项异常              → "可正常训练，关注X" (中置信度)
  - 2项异常              → "建议恢复训练" (中置信度)
  - 3项+异常 或 TSB<-15  → "建议休息" (高置信度)
  - 数据缺失             → "数据不足，结合主观体感" (低置信度)
```

### C5. 趋势结论生成（`/api/body-trend-summary`）

```
输入:
  - 最近 30 天 body_composition 记录
  - 最近 7 天 wellness 记录
  - 当前训练阶段

生成方式（两种可选）:
  方案 A: 规则模板生成（推荐 v1 先用）
    - 体重变化 + 骨骼肌变化 → 套用结论模板
    - HRV/RHR 异常检测 → 套用恢复模板
    - 组合输出

  方案 B: LLM 生成（P1 阶段考虑）
    - 将数据摘要送 LLM
    - 限定输出格式和语气
```

## E. 风险与约束

| 风险 | 影响 | 缓解 |
|------|------|------|
| Readiness 算法依赖 wellness 数据及时性 | 数据滞后导致建议不准 | 降级状态明确提示 |
| 趋势结论规则模板覆盖不全 | 特殊场景文案不匹配 | 加兜底文案"数据不足" |
| 偏差原因标签需要用户手动选择 | 用户可能不填 | 默认"未标记"，不阻塞流程 |
| Body Battery 字段 Garmin API 不一定返回 | 空值频繁 | 4种空值状态区分 |
| 前端改动量较大（3页同时重构） | 回归风险 | 按 Phase 分批上线，每 Phase 验收 |
