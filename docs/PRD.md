# TrainingEdge v1 PRD — Personal Training Decision Cockpit

> **版本**: v1.1
> **日期**: 2026-03-18
> **当前系统版本**: 0.4.0
> **状态**: 已确认（含商用级重构方案）

---

## 1. 产品定位

### 1.1 产品目标

**产品定义**: TrainingEdge 是面向耐力运动与综合训练人群的 AI 决策辅助系统，基于训练负荷、恢复指标、身体组成与执行反馈，提供个体化训练建议、计划调整与执行跟踪。

**边界声明**: 本系统用于训练管理与运动表现优化，不用于疾病诊断、治疗建议或医学判断。

**核心价值**: 用户每天打开系统，30 秒内至少回答 4 个问题：

1. 今天该不该训练
2. 如果训练，应该练什么类型
3. 最近负荷和恢复是否匹配
4. 身体组成是否朝目标变化

### 1.2 不做什么

本期不做：

- 对外开放注册
- 复杂社交功能
- 医疗诊断
- 专业队级别训练处方引擎
- 多角色权限系统
- App 原生端

### 1.3 产品原则

1. **决策优先**，不是展示优先
2. **趋势优先**，不是单点优先
3. **动作建议优先**，不是指标堆砌优先
4. **个人使用优先**，不做复杂权限和多人协作

### 1.4 设计原则

1. 先给结论，再给指标
2. 只保留对训练决策有帮助的信息
3. 区分原始数据、衍生评分、系统建议
4. 数据异常必须显式提示，不允许假装正常
5. 所有变化值必须标明比较基准（上次 / 7天 / 30天）

### 1.5 页面组织三层结构

所有页面都按这三层组织：

1. **结论层** — 今天 / 本周 / 当前身体状态的明确判断
2. **证据层** — 支撑结论的关键指标，数量少、可解释
3. **动作层** — 用户接下来应该做什么

### 1.6 禁止保留的问题

1. 装饰型总分（如不可解释的"身体评分"）
2. 不可解释的抽象指标
3. 与决策无关的炫技模块（如伪精确肌肉疲劳百分比）
4. 空数据占主位
5. 同一指标跨页面重复轰炸
6. "看起来专业但无法触发行动"的可视化

### 1.7 文案边界

**允许使用**：训练建议、恢复状态、身体组成状态、风险提示、个体化调整

**禁止使用**：诊断、医嘱、治疗、病理判断、康复处方

**页脚统一声明**：本系统用于训练管理与运动表现优化，不替代医疗诊断或治疗建议。如存在伤病、持续不适或异常生理指标，请咨询医生或专业医疗人员。

### 1.8 颜色语义统一

| 颜色 | 语义 |
|------|------|
| 蓝色 | 骑行 / 中性信息 / 导航 |
| 绿色 | 恢复良好 / 正向变化 / 低风险 |
| 黄橙色 | 警惕 / 中风险 / 需要观察 |
| 红色 | 偏离 / 高风险 / 拦截 |
| 紫色 | AI / 力量训练 / 系统能力 |

禁止一个颜色同时承担多个语义。

### 1.9 决策优先级

1. 今日训练建议
2. 恢复状态
3. 周计划执行与偏差
4. 身体组成趋势
5. 原始指标明细

## 2. 用户与使用场景

### 2.1 目标用户

- 第一优先：我自己
- 第二优先：家人 / 朋友（少量、熟人）
- 用户特征：有 Garmin / InBody / 健康数据，有训练计划需求，希望通过数据辅助训练判断

### 2.2 典型使用场景

#### 场景 A：晨间快速判断

用户早上打开系统，查看：
- 今日 readiness
- HRV / 静息心率 / 睡眠是否异常
- 今日计划是否适合执行

输出：正常训练 / 降级为 Z2 恢复 / 建议休息

#### 场景 B：周计划执行

用户查看本周计划：
- 已完成多少
- 是否偏离目标
- 偏差是否需要调整剩余计划

#### 场景 C：身体趋势检查

用户查看过去 30 / 90 天：
- 体重、体脂率、骨骼肌、去脂体重
- 恢复指标趋势

判断是否朝目标（减脂 / 增肌 / 维持表现）发展。

## 4. 页面 PRD

### 4A. 面板页 (`/`)

#### 4A.1 页面目标

让用户在 30 秒内完成"今日训练决策"。

#### 4A.2 页面结构（v1.1 重构）

页面按「结论 → 证据 → 动作」重排，只回答 4 个问题：
1. 今天能不能按计划训练？
2. 本周是否偏离目标？
3. 当前恢复是否需要警惕？
4. 最近执行发生了什么值得注意的变化？

##### 模块 1：今日训练决策卡（唯一主卡，第一屏）

- **状态**: 正常训练 / 降级一档 / 恢复训练 / 完全休息
- **置信度**: 高 / 中 / 低（必须附原因，如"缺少 HRV 连续趋势"）
- **建议动作**: 按计划执行 / 主观疲劳高则降级一档 / 今日不加练
- **关键证据**: 只显示 3 条（TSB + 睡眠 + HRV 或 RHR）
- 支持展开查看完整判断依据（评分表）

示例：
```
今日建议：可正常训练
依据：TSB -5.5，存在一定疲劳，但睡眠充足，静息心率稳定
动作：按计划执行；若主观疲劳明显，强度降 1 档
置信度：中（缺少 HRV 连续趋势）
```

##### 模块 2：本周执行偏差（紧跟主卡）

- 本周已完成训练次数 / 计划次数
- 本周已完成 TSS / 计划 TSS
- 偏差判断句（不只是进度条）
- 跳转训练计划页链接

示例判断句：
> 本周进度明显滞后。若明日后仍未补足，建议下调本周剩余训练目标，而非压缩恢复连续补课。

##### 模块 3：恢复证据卡（最多 4 张）

只保留：
1. 睡眠（时长 + 评分）
2. HRV
3. 静息心率
4. TSB

**删除**：7日训练次数卡。

每张卡显示：当前值 + vs 7日均值变化 + 数据时间 + 来源 + 异常高亮。
点击 HRV / 睡眠 / RHR → 跳转身体数据页对应趋势图。

##### 模块 4：近 7 日训练总览（单张综合卡）

替代原"7日训练次数"卡，内容：
- 总时长（如 11.2h）
- 总 TSS（如 406）
- 高强度天数（如 2 天）
- 运动类型分布（🚴4 🏃1 💪4）

##### 模块 5：训练负荷趋势图（证据区）

图上方新增自动结论句：
> 近 30 天 CTL 缓慢回升，近期 ATL 波动加大，TSB 偏低，说明训练刺激增加但恢复压力也在上升。

保留 30/90/180 天切换 + hover 显示 daily TSS + 标记今天。

##### 模块 6：最近活动（复盘入口）

每条活动增加：
- 是否完成计划
- 是否触发偏差
- 是否影响本周安排

#### 4A.3 页面状态

| 状态 | 展示 |
|------|------|
| 空状态 | 请先同步 Garmin 活动与健康数据 |
| 异常状态 | 今日建议不可用：健康数据缺失 / 今日未完成同步 |
| 降级状态 | 数据不足，请结合主观体感判断（不输出确定结论） |

### 4B. 身体数据页 (`/body-data`)

#### 4B.1 页面目标

帮助用户判断：

- 身体组成是否朝目标方向变化
- 当前恢复状态是否支持训练
- 近期波动是真变化还是噪音

#### 4B.2 页面结构（v1.1 重构）

页面目标重定义：不再展示"健康感很强的总览"，而是回答：
1. 身体组成是否朝训练目标前进？
2. 恢复相关生理指标是否支持当前负荷？
3. 近期是否需要营养 / 恢复 / 训练微调？

##### 模块 1：身体组成状态卡（替代原"身体评分"）

**删除**：身体评分主卡（不可解释的抽象总分）。

**新增**：身体组成状态卡，格式为「观察 / 解释 / 动作」三段式：

```
身体组成状态：偏向增肌回补

观察：10 天内体重 +1.7kg，骨骼肌 +1.1kg，体脂率 -0.5%
解释：更接近糖原/水分回补与训练适应，不是单纯增脂
动作：继续当前训练，不建议因短期体重上升主动减量
```

状态标签（5 类）：
1. 偏向增肌回补
2. 稳定维持
3. 轻度减脂
4. 疑似能量不足
5. 需关注异常波动

后面跟恢复状态和训练含义两段，同样「观察 / 解释 / 动作」格式。

##### 模块 2：KPI 卡片（6 张）

只保留高决策价值：
1. 体重（vs 上次 + vs 30 天）
2. 体脂率
3. 骨骼肌（附一句意义说明）
4. HRV
5. 睡眠
6. 静息心率

**删除**：身体评分卡、Body Battery 空卡。

Body Battery 处理规则：
- 有数据：正常显示
- 无数据：**隐藏**（不占主位）

##### 模块 3：InBody 详细数据（降级为证据区）

**改名**：从"InBody 身体成分分析"改为"身体组成明细（数据来源：InBody）"。

加说明：「该区域用于查看明细，不直接决定训练计划；训练建议以趋势和恢复状态综合判断。」

保留当前表格，加 3 列解释信息（重要性 / 稳定性 / 易受干扰因素）。

##### 模块 4：趋势图区域

拆分为 4 张独立图表：
1. 体重趋势
2. 体脂率 + 骨骼肌趋势
3. HRV + 静息心率趋势
4. 睡眠趋势

**每张图上方自动生成一句结论**，不让用户自己猜。

示例：
> 近阶段体重回升主要来自骨骼肌与体内储备恢复，当前未见明显增脂风险。

每张图支持 7天/30天/90天/全部 + 均线开关 + hover 明细。

##### 模块 5：数据来源与可信度

保留，显示来源、最新同步时间、可信度、可比性。

##### 模块 6：InBody 照片识别

保留，增加识别后校验（预览 → 确认 → 入库）。

#### 4B.3 页面文案原则

避免过度确定判断：
- ❌ "你长脂肪了" / "你恢复很好"
- ✅ "当前变化更像……" / "暂未见明显恢复异常" / "结合近期训练负荷，较可能是……"

### 4C. 训练计划页 (`/plan`)

#### 4C.1 页面目标

回答两个问题：

1. 这周该怎么练
2. 计划是否需要因现实执行情况而调整

#### 4C.2 页面结构（v1.1 重构）

页面重定义：训练计划页不是"周日历页"，而是**本周训练执行控制台**。

回答：
1. 这周是什么周？
2. 这周为什么这么排？
3. 今天和接下来该怎么执行？
4. 如果偏离了，系统会怎么处理？

**页面模块顺序（非常重要）**：

##### 模块 1：本周结论条（新增主模块）

- 周类型：Build / Recovery / Maintain
- 本周目标：提升阈值 / 提升耐力 / 恢复吸收
- 预计总时长 / 预计总 TSS
- 骑 / 跑 / 力量次数
- 本周最大约束

示例：
```
本周类型：构建期 Build
主目标：提升阈值能力，维持跑步基础
计划：11.2h / 406 TSS / 骑行4 跑步1 力量4
约束：避免连续 3 天高负荷；强度骑后次日不安排腿部大重量
```

##### 模块 2：今日建议（缩短版，快速扫完）

```
今日：可正常训练
建议：按计划执行
特别注意：若主观疲劳高，降级 1 档
原因：TSB 偏低但恢复未见异常
```

##### 模块 3：周课表（主体）

保留 7 天日历布局。每张训练卡改为：
- **两个标签**：角色标签（关键课/恢复/辅助） + 项目标签（骑行/跑步/力量）
- **可替代性**：不可跳过 / 可降级 / 可延后 / 可取消
- **处理结果**（已完成外）：未完成已取消补做 / 降级补做 / 已转移 / 已顺延

训练卡优先显示：名称 → 时长 → TSS → Zone → 标签 → 处理状态。减少描述文字长度。

##### 模块 4：本周偏差与处理

- 完成率 / TSS 进度 / 偏差判断
- 偏差原因标签：没时间 / 疲劳 / 生病 / 临时替换 / 主动加练
- 系统建议处理：继续原计划 / 压缩后半周 / 降负荷 / 顺延下周 / 进入恢复周

示例：
> 当前偏差：明显滞后。建议：不建议连续补课；优先保留周六关键 VO2 训练，其余课按恢复情况降级。

##### 模块 5：下周预告

- 周类型 / 预计总时长 / 预计总 TSS
- 下周关键课方向
- 风险提示（如有）

##### 模块 6：AI 决策依据（**默认折叠**）

标题："查看本周生成依据"。展开后显示：
- 阶段判断 / 负荷依据 / 恢复依据 / 约束条件

##### 模块 7：恢复约束（替代重复 TSB 展示）

```
当前负荷状态：TSB -5.5，处于可训练但需注意恢复区间
建议：正常训练，不额外加练；若连续两日主观疲劳升高，则下调一档
```

##### 模块 8：本周约束满足情况（替代原"合规追踪 0/10"）

改为清单格式：
- ✅ 周一休息：已满足
- ❌ 每周至少 1 次跑步：未完成
- 🔄 每周 3-4 次骑行：进行中
- ✅ 避免连续 3 天高负荷：当前满足

##### 已删除模块

- **肌肉疲劳热力图**：删除。原因：计算逻辑依赖经验映射不够稳健，用户易误读为身体真实疲劳测量，削弱可信度。
- 替代方案：如需局部恢复约束，改为"训练约束提示"文字形式（如"下肢恢复优先，今日避免腿部大重量"）。

##### 计划操作区

- 生成本周计划
- 重新生成本周
- **标记特殊情况**（生病 / 出差 / 睡眠差）— 优先级高于重新生成
- 一键降级本周
- 复制上周结构

#### 4C.3 计划生成逻辑要求

**输入**:
- 本周可训练时间
- 当前主项
- 目标方向
- 当前阶段
- 最近 7/28 天训练负荷
- wellness
- 身体数据趋势
- 特殊限制

**输出**必须包含:
- 本周结论
- 每日训练安排
- 计划总 TSS
- 风险提示
- 调整建议

### 4D. 活动详情页 AI 骑行复盘模块 (`/activity/{id}`)

#### 4D.1 模块目标

在单次骑行活动详情页中，为用户提供：
1. 对本次骑行的简洁总结
2. 对执行质量和训练价值的结构化判断
3. 对后续训练安排的动作建议
4. 可追溯、可复用、可供 TG 追问的统一分析结果

**定位**：活动级"智能解释层"，位于原始指标区之后。

**不做**：医疗诊断、伤病判断、自由散文式点评、无依据的意志评价、直接修改计划。

#### 4D.2 默认展示：AI 复盘摘要卡

默认只显示 3 部分：

1. **总评**：如"本次为有效耐力骑行，整体执行稳定"
2. **3 条关键判断**：如"有氧负荷达标 / 高强度刺激有限 / 后半程输出略降"
3. **对后续训练影响**：如"明日可按计划训练；若主观疲劳高，降级1档"

按钮：`[展开完整分析]` `[在 TG 继续追问]` `[重新分析]`

#### 4D.3 展开后：六段式完整复盘模板

1. **训练类型识别** — 耐力骑 / 阈值骑 / 混合骑 / 恢复骑 / 比赛型输出
2. **执行质量** — 是否符合计划，偏差在哪
3. **生理成本** — 对恢复和疲劳的影响
4. **能力信号** — 是否支持 eFTP / PDC / 耐力能力更新
5. **异常与干扰因素** — 补给/热应激/地形/设备/停车
6. **后续建议** — 是否改动明日/本周计划

#### 4D.4 结构化存储

AI 复盘结果写入 activity 对象 `ai_review` 字段，不只保存文案：

```json
{

  "analysis_version": "ride_review_v1",
  "generated_at": "2026-03-17T21:35:00+08:00",
  "summary": {
    "overall_label": "有效耐力骑行",
    "one_line_summary": "本次为有效耐力骑行，整体执行稳定。",
    "completion_status": "on_target",
    "fatigue_impact": "moderate",
```

#### 4D.5 TG 交互

TG 不是独立分析器，而是活动 AI 复盘的**对话入口与追问层**：
- 一级回答：读取已有摘要（总评+判断+建议）
- 二级追问：从 narrative 展开
- 三级深挖：跨活动对比、触发深度分析

TG 与活动详情页**必须共用同一份 ai_review**，不生成第二套结论。

#### 4D.6 活动列表 AI 标签

极简标签，如：`[有效耐力骑]` `[完成不足]` `[高疲劳成本]` `[恢复骑]`

#### 4D.7 触发时机

- **自动生成**：活动同步 + 特征提取完成 + 数据完整度达标 + 运动类型为骑行
- **延迟生成**：需计划匹配时先 pending，比对完成后再生成
- **手动重算**：用户点击"重新分析"
- **不生成/降级**：数据缺失严重、设备异常、时长过短、通勤/遛车

#### 4D.8 异常风控

数据缺失或异常时降级生成，提示：
> 当前活动存在异常恢复或数据不完整情况，本次复盘仅供训练参考；如伴随持续不适，请减少训练并由人工或医生复核。

#### 4D.9 API

- `GET /api/activities/{id}/ai-review` — 获取完整复盘
- `POST /api/activities/{id}/ai-review/regenerate` — 手动重算
- `GET /api/activities/{id}/ai-review/summary` — TG 用摘要

## 5. 核心算法展示要求

### 5.1 Readiness 展示分层

不追求"医学正确"，只追求"训练决策可用"。

展示逻辑必须区分 3 层：

| 层级 | 内容 |
|------|------|
| 原始数据 | HRV, RHR, 睡眠, TSB |
| 系统评分 | readiness score, 恢复等级 |
| 最终建议 | 可执行关键课 / 正常训练 / 恢复训练 / 休息 |

**不允许把原始数据和结论混为一谈。**

---

## 6. 数据口径要求

### 6.1 所有变化值必须标明比较基准

- `65.0kg（vs 上次 +1.7kg）`
- `11.7%（vs 30天均值 -0.5%）`

### 6.2 所有页面都要显示数据时间

- `HRV：2026-03-17`
- `InBody：2026-03-14`

### 6.3 多源数据必须显示来源

- Garmin / InBody / OCR / 手工录入

### 6.4 对可疑数据必须做提示

- "本次体脂率与过去趋势差异较大，可能受测量时段或水分影响"

### 6.5 训练完成率口径统一

必须明确区分：

| 口径 | 定义 | 是否包含力量 |
|------|------|-------------|
| `primary_sessions` | 主训练单元（骑行/跑步） | 否 |
| `total_sessions` | 所有训练单元 | 是 |
| `strength_sessions` | 力量训练单元 | 仅力量 |
| `planned_week_tss` | 本周计划 TSS | 不含力量估算 |

前端主显示：主训练完成率 + 本周总负荷进度。力量放在明细中，不与主训练混为一个完成率。

### 6.6 置信度必须为多因子数组

`confidence_reasons` 为 `string[]`，不写死为单一原因。

### 6.7 数据来源表必须包含"决策参与"字段

`decision_role`: `"primary"` | `"auxiliary"` | `"none"`

### 6.8 异常场景人工复核规则

出现以下任一情况时，页面显示人工复核提示而不是继续给强结论：

| 触发条件 | 标识 |
|---------|------|
| 静息心率连续 3 日偏高 (>7日均值+1SD) | `rhr_elevated_3d` |
| HRV 连续 5 日下降趋势 | `hrv_declining_5d` |
| 睡眠连续 3 日 <6h | `sleep_deficit_3d` |
| 用户主动标记伤病/生病/疼痛 | `user_marked_sick` |
| 关键指标缺失 ≥3 项 | `data_insufficient` |

提示文案：
> 当前存在异常恢复或数据不足情况，建议减少训练强度并由人工复核，不建议仅依据系统自动建议继续推进训练。

## 11. 技术约束

### 11.1 开发方式

1. 先输出页面信息架构
2. 再输出 ASCII 线框图
3. 再确认组件拆分
4. 再开发
5. 最后给出：变更说明、风险点、测试清单、回滚方案

### 11.2 技术要求

- 尽量复用现有 API（`/api/summary`, `/api/wellness`, `/api/fitness`, `/api/body-composition` 等）
- 尽量不改底层指标计算（`engine/metrics.py`）
- 优先改模板层 / 接口聚合层 / 页面组件
- 不要先大规模重构数据库

### 11.3 输出物要求

最终至少给出：
- 更新后的页面结构说明
- 修改过的模板文件列表
- 新增/修改 API 列表
- 验收测试步骤
- 已知风险
- 后续建议

## 12. 关键假设

1. 这个系统主要供自己使用，不是正式商用产品
2. 更需要"辅助判断"，不是"自动替你训练"
3. 当前能接受用现有数据源先做 80 分方案
4. 这一版开发应优先前端产品化，而不是后端算法大改

---

## 13. 注意事项

### 13.1 不要把"数据多"误当成"决策强"

最大问题不是缺字段，而是缺"结论层"和"动作层"。

### 13.2 不要把"AI计划页"继续做成更复杂的展示页

训练计划页最重要的是：可执行、可调整、可解释。不是再多放几个指标。

---

## 14. 失效条件

这份 PRD 在以下情况下要重写：

1. 决定把产品改成对外开放使用
2. 准备接入更多家庭成员并做账号体系
3. 决定把重点转到移动端 App
4. 打算重写数据模型而不是基于现有系统迭代
5. 发现最常用的其实只是"活动详情分析"，而不是训练决策
